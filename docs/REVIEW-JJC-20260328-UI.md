# 门下省审查意见 · chip-sim UI 重做实施规划

**审查人：** 门下省·侍中  
**审查日期：** 2026-03-28  
**审查对象：** `/chip-sim/docs/IMPLEMENTATION-PLAN.md`  
**评分： B**  
**结论： 有条件通过**

---

## 一、总评

规划方向正确——数据驱动芯片配置、统一 WebGL 渲染、分阶段实施——这些思路合理。但存在 **3 个严重问题** 和 **5 个中等问题**，需修改后方可进入中书省起草环节。

---

## 二、严重问题（必须修改）

### S1. "Silicone Float UI" 不存在

规划全文引用 "Silicone Float UI" 作为 UI 风格和 CSS 变量体系的来源。经核实：
- npm 上无此包
- 项目现有代码中从未引用
- 互联网无可查证的开源项目

**问题：** 如果这是一个内部约定名称，需要附上设计规范文档或 Figma 链接。如果是要参考某个现有框架（如 Linear、Vercel、Raycast 的设计语言），应明确写出。

**建议：** 改为明确的技术方案：
- 方案 A：基于现有 `theme.css` + `ThemeSystem.ts` 迭代，已有完整暗色/亮色主题体系
- 方案 B：引入 shadcn/ui（Radix + Tailwind），业界成熟
- 方案 C：自定义设计系统，但需提供设计规范文档

### S2. 删除清单过于激进，缺乏复用分析

规划第三章列出 9 个文件/目录删除，但现有代码包含大量可复用逻辑：

| 被标记删除的文件 | 实际可复用内容 |
|---|---|
| `circuit-store.ts`（1854 行） | undo/redo、项目管理、自动保存、WebSocket 集成——这些是核心业务逻辑，不应删除 |
| `EditorPage.tsx`（615 行） | 响应式布局逻辑、Tab 管理、快捷键绑定——可重构而非删除 |
| `simulator/*` | `SimulationBridge.tsx` 与后端 WebSocket 通信，删除后需完全重写 |
| `WaveformPanel.tsx` | 波形渲染已有完整实现 |
| `CircuitCanvas.tsx` | 与 WebGL 渲染器的桥接层 |

**问题：** 项目有 ~200 个源文件、21 个测试文件、827 行 WebGL 渲染器。"删了重写"策略会导致大量已验证的功能丢失。

**建议：** 改为 **渐进式重构**：
1. 先新建 UI 组件层（`ui/` 目录）→ 不影响现有功能
2. 用新 UI 组件逐步替换 EditorPage 中的旧组件
3. WebGL 渲染器在现有 `webgl-renderer.ts` 上增量改造
4. 电路 store 保留核心逻辑，只扩展 MCU 相关状态

### S3. C 代码解析器范围过大，无技术验证

Phase 4 要求解析：
- C51: `P1 = 0xFF;`, `TMOD = 0x01;`
- STM32: `HAL_GPIO_WritePin()`, `HAL_UART_Transmit()`
- ESP32: `digitalWrite()`, `Serial.begin()`

**问题：** 完整的 C 代码解析器（即使是子集）是一个编译器前端项目，涉及词法分析、语法分析、语义分析。用正则或简单模式匹配只能处理最简单的情况，一旦用户写循环、条件分支、函数调用就会失败。

**建议：**
- 最低限度：只支持"粘贴一行执行一行"的简单模式匹配
- 中等方案：调用 WebAssembly 版的 GCC/Clang 前端做 AST 解析
- 或者明确说明：**V1 只支持预设示例代码模板**，自定义代码解析放到 V2

## 三、中等问题（建议修改）

### M1. 时间估算不精确

"约 3 轮"、"约 2 轮" 不是可度量的时间单位。1 轮 = 1 天？1 周？1 个 sprint？

**建议：** 改为小时或天数估算，并标注置信度。例如：
- Phase 1：5-8 个工作日（高置信度）
- Phase 2：8-12 个工作日（中置信度，WebGL 交互调试不可预测）

### M2. 后端变更未提及

现有 Go 后端已有完整的 MCU 仿真引擎（`mcu_sim.go`、`gpio.go`、`adc.go`、`protocols/` 等），但规划中完全没有提到后端需要做哪些调整：
- 芯片 JSON 配置是否需要后端加载？
- 外设行为模型是前端纯计算还是后端参与？
- WebSocket 协议是否需要变更？

**建议：** 增加"后端变更"章节，明确前后端职责边界。

### M3. Phase 顺序有依赖风险

Phase 2（WebGL 画布 + 交互）排在 Phase 3（芯片系统）之前，但芯片引脚图渲染依赖芯片配置。画布做完后，芯片渲染可能需要改画布 API。

**建议：** 调整为：
- Phase 1：UI 基础 + 主页面布局
- Phase 2：芯片 JSON 配置 + chip-loader（数据先行）
- Phase 3：WebGL 画布 + 交互 + 芯片渲染
- Phase 4：外设模型 + 代码解析
- Phase 5：面板完善
- Phase 6：导出 + 打磨

### M4. 测试策略缺失

现有项目有 21 个测试文件，但规划中没有提到：
- 哪些现有测试需要保留/修改
- 新功能的测试覆盖率目标
- E2E 测试计划（画布交互很难单元测试）

**建议：** 每个 Phase 验收标准中增加测试要求。

### M5. 迁移和兼容性

用户已有项目数据（localStorage 中的电路），规划没有提到：
- 新版是否兼容旧数据格式
- 是否需要数据迁移脚本
- `circuit.ts` 类型定义是否变更

**建议：** 增加"数据迁移"章节，或明确声明 V1 不兼容旧数据。

## 四、肯定之处

1. **芯片 JSON 配置方案优秀** —— 数据驱动、易于扩展新型号，JSON 结构设计合理
2. **WebGL 唯一渲染层** —— 正确的方向，避免 Canvas 2D / WebGL 双维护
3. **Phase 1-6 分阶段验收** —— 整体结构清晰
4. **交互设计详细** —— 手型/拖拽/右键/双击等交互规范写得清楚
5. **保留 Zustand** —— 技术栈延续性好

## 五、现有代码复用评估

| 模块 | 现有代码 | 可复用度 | 说明 |
|---|---|---|---|
| WebGL 渲染器 | `webgl-renderer.ts`（827行） | ⭐⭐⭐⭐ | 基础框架可直接复用，扩展元件渲染即可 |
| 状态管理 | `circuit-store.ts`（1854行） | ⭐⭐⭐⭐⭐ | 核心逻辑完善，只需扩展 MCU 相关状态 |
| 元件库 | `core/components/`（20+文件） | ⭐⭐⭐⭐⭐ | 行为模型已实现，无需重写 |
| MCU 仿真 | `core/mcu/` + Go 后端 | ⭐⭐⭐⭐ | 前后端已有时钟/中断/定时器/GPIO，需扩展 |
| 主题系统 | `theme.css` + `ThemeSystem.ts` | ⭐⭐⭐⭐ | 完整的暗色/亮色体系，可直接迭代 |
| 测试 | 21 个测试文件 | ⭐⭐⭐ | 需适配新 API，但逻辑可复用 |
| 编辑器页面 | `EditorPage.tsx` | ⭐⭐⭐ | 布局逻辑可复用，UI 需替换 |
| 波形面板 | `WaveformPanel.tsx` | ⭐⭐⭐ | 渲染逻辑可复用，UI 需替换 |

**结论：** 现有代码可复用约 **60-70%** 的核心逻辑，不应采用"删除重写"策略。

## 六、风险提示

| 风险 | 等级 | 说明 |
|---|---|---|
| WebGL 画布交互调试 | 🔴 高 | 平移/缩放/拖拽/连线的边界情况极多，预计 Phase 2 会超时 |
| C 代码解析器 | 🔴 高 | 如不限制范围，可能成为项目瓶颈 |
| "Silicone Float UI" 设计一致性 | 🟡 中 | 如果是自定义设计系统，需要设计师参与 |
| 16 个 UI 组件一次性实现 | 🟡 中 | 建议先做 6 个核心组件（Button/Panel/Input/Tabs/Modal/Toast），其余按需补充 |
| 后端 WebSocket 协议变更 | 🟡 中 | 如果前后端职责重新划分，需同步修改 |

## 八、结论

**有条件通过。** 规划整体方向正确，芯片 JSON 配置和 WebGL 统一渲染是亮点。但"删除重写"策略过于激进，"Silicone Float UI" 缺乏依据，代码解析器范围需收窄。修改上述 S1-S3 后，可提交中书省进入起草阶段。
